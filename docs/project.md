# 프로젝트 개요

## 목적

KHU:DArchive는 사용자의 이력서, 자기소개서, 포트폴리오 메모, 프로젝트 회고 등 흩어진 커리어 기록을 경험 단위로 구조화하고, JD/문항별로 적합한 경험을 추천하는 RAG 기반 경험 저장소 프로젝트입니다.

사용자의 과거 기록을 검색 가능한 경험 Vault로 만들고, 사용자가 입력한 JD나 자기소개서 문항에 대해 어떤 경험을 활용하면 좋을지 추천하는 데 집중합니다.

---

## 핵심 문제

사용자는 자기소개서를 작성할 때 과거 경험을 다시 찾고, 정리하고, 문항에 맞는 경험을 고르는 데 많은 시간을 사용합니다.

특히 다음과 같은 문제가 있습니다.

* 과거 경험이 이력서, 자기소개서, 포트폴리오, 메모 등에 흩어져 있음
* 같은 경험을 여러 번 반복해서 정리해야 함
* 어떤 경험이 특정 JD나 문항에 적합한지 판단하기 어려움
* 경험의 성과, 역할, 수치, 기술적 행동이 부족하게 기록된 경우가 많음
* 자기소개서 작성 전에 경험 자체를 정리하는 과정이 부족함

KHU:DArchive는 이 문제를 해결하기 위해 사용자의 과거 기록을 경험 Vault로 만들고, 문항/JD 기준으로 적합한 경험을 추천합니다.

---

## 핵심 기능

### 1. 과거 기록 입력

사용자는 과거 자기소개서, 이력서, 포트폴리오 메모, 프로젝트 회고 등을 입력합니다.

텍스트 입력을 우선 지원하고, 이후 파일 업로드, Notion 연동, 웹훅 기반 자동 아카이빙으로 확장할 수 있습니다.

### 2. 원문 기록 보존

사용자가 입력한 원문은 그대로 저장합니다.

AI가 추출한 경험 정보가 잘못되었을 때 원문으로 다시 확인할 수 있어야 하기 때문입니다.
또한 추천 결과에서 “어떤 기록을 근거로 추천했는지” 보여주기 위해 원문 출처를 보존합니다.

### 3. 경험 단위 구조화

입력된 기록을 AI가 경험 단위로 분해합니다.

각 경험은 다음 정보를 포함하는 경험 카드로 저장됩니다.

* 경험 제목
* 요약
* 경험 유형
* 기간
* 조직 또는 활동명
* 본인 역할
* 상황
* 과제
* 행동
* 결과
* 배운 점
* 기술 스택
* 역량
* 키워드
* 출처
* 완성도 점수

### 4. 경험별 출처 추적

각 경험이 어떤 원문에서 추출되었는지 추적합니다.

예를 들어 하나의 경험은 다음과 같은 여러 출처를 가질 수 있습니다.

* 2025 이력서 프로젝트 섹션
* 카카오 자기소개서 1번 문항
* 프로젝트 회고 메모
* 포트폴리오 문서

출처 추적을 통해 추천 결과의 신뢰도를 높입니다.

### 5. 검색 가능한 경험 Vault 구축

구조화된 경험은 RAG 검색을 위해 chunk로 나누고 embedding을 생성합니다.

검색은 chunk 단위로 수행하지만, 최종 결과는 experience 단위로 묶어 보여줍니다.

즉, 사용자는 단순 문장 검색 결과가 아니라 “어떤 경험을 활용해야 하는지”를 확인할 수 있습니다.

### 6. JD/문항 기반 경험 추천

사용자가 지원 직무, JD, 자기소개서 문항을 입력하면 시스템은 다음 과정을 수행합니다.

1. JD/문항의 의도와 요구 역량을 분석합니다.
2. 경험 Vault에서 관련 chunk를 검색합니다.
3. 검색된 chunk를 experience 단위로 그룹핑합니다.
4. LLM이 각 경험의 적합도를 판단합니다.
5. 추천/보류/비추천 경험과 근거를 제공합니다.

추천 결과에는 다음 정보가 포함됩니다.

* 추천 경험
* 추천 이유
* 원문 근거
* 관련 기술과 역량
* 부족한 정보
* 보완 질문

### 7. 부족한 정보 보완

경험에 부족한 정보가 있으면 보완 질문을 생성합니다.

예를 들어 다음과 같은 정보를 질문합니다.

* 본인의 구체적인 역할
* 성과 수치
* 문제 상황
* 기술적 행동
* 협업 또는 갈등 상황
* 결과와 배운 점

사용자가 답변하면 해당 경험 카드, chunk, embedding을 다시 업데이트합니다.
이를 통해 경험 Vault의 품질이 점점 좋아집니다.

### 8. OpenAPI 기반 API 명세 관리

API는 FastAPI route와 Pydantic schema를 기준으로 관리합니다.

FastAPI가 생성하는 OpenAPI 문서를 팀 간 API 계약 문서로 사용합니다.

## 화면 구성

프론트엔드 화면은 다음 흐름을 기준으로 설계합니다.

### 1. 기록 입력 화면

사용자가 과거 자기소개서, 이력서, 프로젝트 회고, 포트폴리오 메모 등을 입력합니다.

### 2. 처리 결과 화면

AI가 추출한 경험 목록과 각 경험의 완성도, 부족한 정보를 확인합니다.

### 3. 경험 Vault 화면

저장된 경험 카드 목록을 확인합니다.

각 경험의 제목, 요약, 기술 스택, 역량, 완성도 점수를 보여줍니다.

### 4. 경험 상세 화면

특정 경험의 상세 정보를 확인합니다.

다음 정보를 보여줍니다.

* STAR 구조
* 기술 스택
* 역량
* 원문 출처
* 원문 근거
* 부족한 정보
* 보완 질문

### 5. JD/문항 입력 화면

사용자가 지원 직무, JD, 자기소개서 문항을 입력합니다.

### 6. 경험 추천 결과 화면

시스템이 추천/보류/비추천 경험을 보여줍니다.

각 경험에 대해 다음 정보를 제공합니다.

* 추천 여부
* 추천 이유
* 원문 근거
* JD/문항과 연결되는 역량
* 부족한 정보
* 보완 질문

### 7. 보완 질문 답변 화면

사용자가 부족한 정보에 답변합니다.

답변 내용은 경험 Vault에 반영되고, 이후 추천 품질을 높이는 데 사용됩니다.

### 8. 경험 검색 화면

사용자가 직무, 역량, 기술 키워드로 저장된 경험을 탐색합니다.

예를 들어 “백엔드 성능 개선 경험”, “협업 경험”, “데이터 분석 경험”처럼 검색할 수 있습니다.

## 시스템 흐름

```text
[과거 기록 입력]
이력서 / 자기소개서 / 회고 / 포트폴리오 메모
        ↓
[경험 Vault 구축]
원문 저장 → 텍스트 정리 → 경험 추출 → 출처 연결 → chunk 생성 → embedding 저장
        ↓
[경험 Vault]
experience / source evidence / chunk / embedding / question
        ↓
[JD·문항 입력]
지원 직무 / JD / 자기소개서 문항
        ↓
[Query 분석]
문항 의도 / 요구 역량 / 기술 스택 / 경험 선택 기준 추출
        ↓
[Retrieval]
관련 chunk 검색 → experience 단위 그룹핑
        ↓
[LLM Reasoning]
추천 / 보류 / 비추천 판단
        ↓
[결과 출력]
경험 추천 / 추천 근거 / 부족한 정보 / 보완 질문
        ↓
[사용자 보완]
답변 저장 → 경험 카드 업데이트 → chunk·embedding 재생성
```

## 역할 구분

프로젝트는 크게 세 파트로 나눌 수 있습니다.

### 1. 경험 Vault 구축 파트

사용자의 과거 기록을 저장하고, AI로 경험 단위로 구조화합니다.

주요 작업:

* 원문 문서 저장
* 텍스트 정리
* 경험 추출
* 경험 카드 저장
* 출처 연결
* chunk 생성
* embedding 저장
* 부족 정보 질문 생성

### 2. RAG 기반 경험 추천 파트

사용자가 입력한 JD나 문항에 맞는 경험을 추천합니다.

주요 작업:

* JD/문항 분석
* retrieval query 생성
* 관련 chunk 검색
* experience 단위 그룹핑
* LLM reasoning
* 추천/보류/비추천 판단
* 추천 근거와 부족 정보 생성

### 3. 프론트엔드 파트

사용자가 서비스를 실제로 사용할 수 있는 화면을 구현합니다.

주요 작업:

* 기록 입력 화면
* 경험 목록 화면
* 경험 상세 화면
* JD/문항 입력 화면
* 경험 추천 결과 화면
* 보완 질문 답변 화면
* API 연동

---

## 최종 목표

KHU:DArchive의 최종 목표는 사용자가 자기소개서를 작성하기 전에 자신의 경험을 쉽게 찾고, 정리하고, 문항에 맞게 선택할 수 있도록 돕는 것입니다.

이를 통해 사용자는 과거 기록을 반복해서 뒤지지 않고, 자신의 경험을 근거와 함께 재사용할 수 있습니다.
