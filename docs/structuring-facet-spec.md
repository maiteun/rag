# 경험 구조화 — facet 추출 프롬프트 & 설계 (공유용)

> 대상: 빌드타임(경험 구조화) 담당
> 보낸 사람: RAG 파트
> 목적: 경험을 **"역량(강점) 단위 facet + 세부 아카이빙"** 으로 구조화하는 방식 제안. 아래 프롬프트/스키마를 그대로 쓰거나 아이디어로 발전시켜 주세요.
> ※ 예시는 전부 가상의 제너릭 사례입니다.

---

## 1. 왜 이렇게 하나 (배경)

현재 구조화는 강점을 납작한 태그(`skill: 팀코디네이터`)로 뽑는데, 이러면:
- **검색(RAG)**: "협업 경험" 문항이 와도 태그만으론 근거를 못 꺼내고, 서사가 없어 임베딩 매칭도 나쁨.
- **생성(자소서)**: "~역할을 했습니다"처럼 근거 없이 밋밋해지거나 지어내게 됨(과장 금지 위반).

그래서 **① 매칭 가능한 역량 태그 + ② 원문 인과 서사 보존 + ③ 세부는 아카이빙**의 3박자가 필요합니다.

## 2. 핵심 원칙

1. **역량 단위 태깅.** facet은 '무슨 일을 했나(내용)'가 아니라 '이 일이 지원자의 **어떤 역량/강점**을 증명하나'로 태깅. (`-99 보간` → capability=`데이터 전처리`)
2. **주도성 부각.** 지원자가 직접 제안·주도·결정한 부분을 명시. 그게 강점의 핵심.
3. **입도 2단 (grain).** 표면 facet = 강점 단위(자소서에 꺼내 쓰는 것). 그 아래 `details[]` = 도구·수치·기법 같은 **세부는 아카이빙**(사라지지 않고 붙어있되 표면을 어지럽히지 않음). 예: "실시간 시스템 구현"이 facet, "40ms 버퍼", "OBS 사용"은 details.
4. **요약 금지·근거 보존.** S/A/R에 원문 디테일(수치·방법·이유)을 그대로 살림. 원문에 없는 건 `null`(→ 역질문). `evidence`에 원문 문장 인용.

## 3. 출력 스키마

```jsonc
{
  "facets": [
    {
      "capability": "데이터 전처리",              // 강점 태그 (JD 요구역량과 매칭되는 단위)
      "theme": "프로젝트 수행",                   // (선택) 굵은 그룹: 협업/프로젝트수행/커뮤니케이션 등. 검색용 아님, 보기·묶기용
      "label": "결측·이상치를 정제해 분석 데이터셋 구축",
      "situation": "…원문 배경/문제를 요약 말고 그대로…",   // 없으면 null
      "action": "…본인이 한 구체적 행동/방법/판단, 주도성 명시…",  // 없으면 null
      "result": "…결과/성과/배움, 수치 포함…",              // 없으면 null
      "details": ["Python으로 3개월치 로그 처리", "결측 5% 제거"],  // 세부 아카이빙 (도구·수치·기법)
      "evidence": ["원문 문장 인용1", "원문 문장 인용2"]           // 근거 (그대로 인용)
    }
  ]
}
```

Pydantic (langchain `with_structured_output`용):

```python
class Facet(BaseModel):
    capability: str        # 증명하는 역량/강점 (짧게, JD 요구역량과 매칭 가능)
    theme: str | None      # (선택) 굵은 그룹 태그
    label: str
    situation: str | None
    action: str | None
    result: str | None
    details: list[str]     # 세부 아카이빙 (도구·수치·기법)
    evidence: list[str]    # 원문 인용

class ExperienceFacets(BaseModel):
    facets: list[Facet]
```

## 4. 시스템 프롬프트 (그대로 사용 가능)

```
너는 자소서 경험 원문을 '역량(강점) 단위 facet'으로 구조화한다.

원칙:
(1) 각 facet은 '무슨 일을 했나'가 아니라 '이 일이 지원자의 어떤 역량·강점을 증명하나'로
    태깅한다(capability). 예: '-99 값 보간'은 capability='데이터 전처리'.
    지원자가 직접 제안·주도·결정한 주도성을 반드시 드러내라.
(2) 절대 요약하지 마라. situation/action/result에 원문의 구체 내용(수치·방법·도구·이유·순서)을
    최대한 보존해 여러 문장으로 풍부하게 써라.
(3) 입도 2단으로 나눠라. 표면 facet은 '강점 단위'(자소서에 꺼내 쓰는 수준)로 잡고,
    도구·수치·기법 같은 지나치게 세부적인 것은 그 facet의 details[]에 넣어 아카이빙하라.
    예: facet '실시간 시스템 구현' + details ['40ms 롤링 버퍼', 'OBS 가상카메라'].
    표면 facet은 강점 단위로 적당히(한 경험에 보통 8~15개), details로 세부를 흡수한다.
(4) theme(선택)으로 굵은 그룹(협업/프로젝트수행/커뮤니케이션 등)을 달아도 된다. 이는 검색이 아니라
    보기·묶기용이다.
(5) 원문에 실제로 있는 내용만. 없으면 null. evidence에 원문 문장을 그대로 인용한다.
    (과장 금지: 원문에 없는 사실을 채우지 말 것.)
```

## 5. 제너릭 예시 (가상 사례)

**원문(가상):**
> "쇼핑몰 이탈률이 높다는 논의 중, 나는 구매 로그를 직접 분석해보자고 제안했다. Python으로 3개월치 로그를 전처리하고(결측 5% 제거) 코호트 분석으로 결제 단계 이탈 지점을 찾았다. 회의록을 정리하고 역할을 나눠 일정을 관리했다. 결제 UI를 개선해 이탈률을 12% 낮췄다."

**before (현재 방식):** `skills: [데이터분석, 협업]`

**after (제안 방식):**

| capability | label | details | (주도성) |
|---|---|---|---|
| 주도적 문제 정의 | 이탈률 문제에 로그 분석 접근을 제안 | — | "직접 제안" 부각 |
| 데이터 전처리 | 구매 로그를 정제해 분석셋 구축 | `Python`, `3개월치`, `결측 5% 제거` | |
| 분석 기법 활용 | 코호트 분석으로 이탈 지점 규명 | `결제 단계 이탈 식별` | |
| 팀 코디네이션 | 회의록·역할분담으로 일정 관리 | — | |
| 성과 창출 | 결제 UI 개선으로 이탈률 12%↓ | `12% 감소` | |

→ 세부(도구·수치)는 details로 접히고, 표면은 매칭 가능한 강점 단위로 정리됨.

## 6. RAG(검색)에서 어떻게 쓰나 (참고)

- **검색은 facet 단위 flat 임베딩.** theme으로 폴더 라우팅하지 않는다(작은 코퍼스 + 교차 facet 때문). theme/capability는 태그로 그룹뷰·소프트 가산에만.
- facet 임베딩에 details를 포함하면 "OBS" 같은 특정 검색도 잡힘. details는 생성 단계에서 근거로도 사용.
- 문항유형(협업/문제해결 등)과 capability를 느슨히 매칭해 관련 강점을 우선 노출.

## 7. 하드룰 (필수)

- **과장 금지**: 원문에 없는 사실 채우지 않기. 비면 `null` → 역질문 트리거.
- **출처 유지**: `evidence`에 원문 인용, 원본 문서 id/위치 함께 저장.
- **병합 보수적**: 새 입력으로 기존 경험을 강화(enrich)할 땐 자동 반영 말고 사용자 확인(false merge 방지).
